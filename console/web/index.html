Engagements

Un engagement est un espace de travail isolé (à la workspace Metasploit) : son propre scope (fail-closed), son propre ledger tamper-evident, ses findings / runs / techniques / workflows. Un engagement ne voit JAMAIS les données d'un autre. On ne peut ni archiver ni supprimer le dernier engagement actif. Création/édition = operator ; archivage/suppression = admin.

Lancement de campagne — gouverné & audité

—

Lancement non-exploit / non-destructif par défaut : les modules exploit / destructif restent grisés tant que l'opt-in gouverné « fort impact » (zone danger ci-dessous) n'est pas activé — il exige armer + raison + secret opérateur et une double-confirmation. Les cibles doivent appartenir au scope serveur : hors-scope = VETO dur, jamais élargi via le web. Chaque lancement est journalisé au ledger (console.run.start).

Armer (--arm) journalise l'intention de tirer les actions FIRE. Seul, l'arm n'autorise PAS l'exploit/le destructif : il faut activer l'opt-in « fort impact » de la zone danger ci-dessous (armer + raison + secret opérateur). Le scope serveur reste dur : hors-scope = VETO, quelle que soit l'autorisation.
zone danger
Action à fort impact, scope-bornée et auditée. L'opt-in n'est honoré que si armer est coché, une raison est renseignée et le secret opérateur est fourni. Le garde-fou de périmètre reste dur : toute cible hors-scope est vétoée côté serveur, sans exception. Chaque lancement à fort impact est journalisé au ledger.

Dry-plan —

Chaque action proposée par le moteur porte son verdict garde-fou : FIRE / DRY_RUN / VETO / SKIP. Coche les actions RECON / non-exploit à approuver puis relance en propose→auto ; l'exploit reste bloqué côté serveur dans cette passe.

Importer un scan — migration Faraday/Trickest/reNgine/Osmedeus

—

Ingère une sortie d'outil déjà produite (nmap XML, nuclei JSON/JSONL, export Burp XML, httpx/ffuf JSON, listes subfinder/amass, ou JSON/CSV générique) en findings Forge orientés preuve : un scanner qui s'auto-déclare (nuclei/burp) devient reported_by_tool, la recon/découverte (nmap/httpx/ffuf/subfinder) devient tested — jamais vulnerable. Pur data, aucune exécution. Les assets hors périmètre sont écartés (scope-guard) et les secrets du fichier sont rédigés. L'import est gouverné (opérateur) et journalisé au ledger.

Capacités & Modules

Catalogue des modules du moteur (lecture /api/modules). Le badge web indique un module autorisé en cadre web ; exploit / destructif portent un risque accru (gatés par les ROE au lancement).

Techniques & Sélection

Catalogue des techniques du moteur, groupé par catégorie (lecture /api/techniques, DÉRIVÉ du registre). « Au scope, retirer des tests automatiques » : le profil donne l'ensemble de base, puis les toggles par catégorie / par technique ajoutent ou retirent (la désactivation prime — fail-closed). Une technique désactivée n'est NI planifiée NI tirée (le moteur l'ENFORCE, en plus du scope-guard). La mutation est réservée operator/admin et journalisée au ledger. Badges : BB éligible bug bounty · pentest pentest-only.

Workflows

Pipelines composés sans code : une sélection ordonnée de techniques/outils (+ params par étape), sauvegardée et éditable — absorbe les scan-engines de reNgine, les workflows d'Osmedeus et les pipelines de Trickest. Un workflow est une PROPOSITION : le scope-guard ROE et la sélection par-scope restent seuls juges — une étape hors-scope / désactivée est larguée (fail-closed). Les étapes exploit restent derrière l'opt-in fort-impact. Les workflows intégré (dérivés du registre) sont non supprimables. Création/édition réservées operator/admin et journalisées au ledger.

Bibliothèque de findings

Modèles de findings réutilisables (titre / description / remédiation / CWE / sévérité) à gabarits paramétrés : les placeholders {target}, {param}… sont remplis à l'application. La bibliothèque est GLOBALE — un modèle est réutilisable d'un engagement à l'autre. Appliquer un modèle crée un finding dans l'engagement ACTIF uniquement (isolation : le finding produit appartient à son engagement, jamais à un autre). Création/édition réservées operator, suppression admin ; chaque action est journalisée au ledger.

Matrice ATT&CK — grille tactique × technique (par engagement)

Vraie grille ATT&CK : colonnes = tactiques du kill-chain, cellules = techniques. Couleur par état : exercée + détectée (a tiré), exercée non détectée (0 tir), grisée = non exercée (couverture manquante). Le MTTD est surfacé sur les cellules détectées quand une source de détection le mesure. Scopé à l'engagement actif.

—

ROE — décisions du garde-fou

Chaque action proposée par le moteur passe le garde-fou : FIRE (exécutée), DRY_RUN (simulée), VETO (bloquée). La transparence anti-masquage : on voit pourquoi (raisons).

Ledger d'engagement

Journal append-only chaîné par SHA-256. La console recalcule la chaîne de hash (intégrité hash-only) mais ne vérifie pas la signature (sig_checked: false — pas de clé côté console). Pour vérifier la signature cryptographique : lancer forge ledger verify --pubkey <clé> en CLI.

Administration — comptes

Comptes individuels (viewer lecture · operator arme les campagnes · admin administre). Chaque mutation est attribuée à votre compte et ledgerisée. La désactivation, la rétrogradation et la réinitialisation de mot de passe révoquent immédiatement les sessions du compte. Le dernier admin activé ne peut être ni désactivé, ni rétrogradé, ni supprimé (garde-fou anti-verrouillage).

chargement…

Tenants enterprise

Un tenant est le sommet de la hiérarchie TENANT → ENGAGEMENT → findings/runs : un espace multi-client isolé (chaque tenant a ses engagements, son ledger tamper-evident, ses données). Un utilisateur d'un tenant ne voit jamais les données d'un autre (filtre fail-closed, deny-by-default). Réservé au platform-admin (admin console ou super-admin) ; chaque action est ledgerisée. Garde-fous : impossible d'archiver le dernier tenant actif ou de retirer le dernier tenant_admin d'un tenant.

chargement…

Identité / SSO enterprise

Fédération d'identité ENTERPRISE : connexion OIDC (SSO), provisioning SCIM 2.0, et RBAC avancé (mapping groupe IdP → rôle Forge + grant tenant). Une identité SSO/SCIM n'obtient que les droits que son mapping de groupe confère (moindre privilège, fail-closed) — jamais super-admin via SSO. Les secrets (client_secret, token SCIM) sont write-only : jamais réaffichés. En build community (flag OFF) : cette vue n'existe pas, comptes locaux inchangés.

Compliance enterprise

WORM / rétention / legal-hold sur le ledger tamper-evident + les données d'engagement, et export de preuves d'audit SOC 2 / ISO 27001. La rétention est la plus spécifique (engagement → tenant → global) ; un legal-hold (à n'importe quel niveau) bloque toute purge/suppression — le hold gagne toujours (fail-closed). Chaque mutation est attribuée et ledgerisée. Les secrets (clé d'archive, endpoint/credential du signer) sont rédigés : jamais affichés. En build community (flag OFF) : cette vue n'existe pas, ledger/données inchangés.

Signer du ledger d'audit (KMS / HSM)

La clé privée Ed25519 du ledger peut vivre hors-hôte (KMS/HSM/exec). La vérification n'utilise que la clé publique ; l'endpoint / le credential / l'argv sont secrets — jamais affichés (seul un booléen *_set l'indique). Configuration en lecture seule (via variables d'environnement FORGE_LEDGER_SIGNER*).

—

Export de preuves (SOC 2 / ISO 27001)

Bundle lecture seule (ne modifie rien) : piste d'audit d'autorisation, état RBAC, journal d'accès/mutation, attestation de sauvegarde (restore-prouvé) et attestation d'intégrité du ledger (head hash + clé publique Ed25519 + résultat de vérification). Isolé par tenant/engagement, secrets rédigés. L'export est lui-même ledgerisé.

Administration — connecteurs

Gouvernance des connecteurs (« installer / désinstaller » opérationnel). État = interrupteur opérateur (enabled) ; override dispo = surcharge de la sonde host (available_override : auto / forcer disponible / forcer indisponible) ; web = lançable depuis le web. Disponibilité effective = enabled ET (override ?? sonde). Un connecteur désactivé ou forcé indisponible est SKIP au tir même si son binaire est présent — y compris pour les modules choisis par le planner. Chaque mutation est attribuée à votre compte et ledgerisée.

Administration — ajouter un outil

Déclarez votre propre outil CLI (ToolSpec déclaratif : binaire + argv_template tokenisé no-shell + params_schema typé + flag_allowlist) — sans éditer de fichier ni recompiler. À l'ajout, l'outil est persisté (dossier server-managed) puis le catalogue est rechargé à chaud : il apparaît dans Capacités et dans Campagne (son formulaire de params est rendu depuis le schéma). Gouvernance identique à un module natif : scope-guard fail-closed, argv fixe no-shell, allowlist de drapeaux, statut jamais « vulnerable », plancher exploit (arm + raison). Admin-only, chaque ajout/suppression est attribué et ledgerisé. Un binaire absent du runtime -> outil « indispo » (skippé au run).

Administration — source de détection —

Source BLUE (SIEM/IDS/pare-feu) corrélée à la boucle purple par identité MITRE. N'importe quelle infra se câble ici sans code (Plume, CrowdSec, FortiGate, pfSense/OPNsense, Elastic/OpenSearch, fichier, commande). Le secret est write-only (affiché ••• une fois posé, jamais renvoyé) et manié comme un secret de session. Une source absente/injoignable ⇒ mesure déclarée impossible (source_reachable:false), jamais de détection inventée. Chaque enregistrement est ledgerisé. Détails : docs/DETECTION.md.

chargement…

Administration — sauvegarde & restauration chiffré

Archive toujours chiffrée (argon2id + XChaCha20-Poly1305) : elle embarque la base, le ledger tamper-evident et la clé de signature .ed25519 — il n'existe aucun chemin en clair. La passphrase est obligatoire (fail-closed) et n'est jamais stockée/loggée/ledgerisée (elle n'est pas conservée côté navigateur au-delà de la requête). La restauration valide + vérifie par défaut (aucune écriture) ; un swap en place exige une confirmation explicite et un redémarrage. Chaque action est ledgerisée (métadonnées seules). Détails : docs/BACKUP.md.

Console Forge gouverné

Lance des sous-commandes forge gouvernées depuis l'UI — plus besoin de docker compose exec forge forge …. Ce n'est pas un shell : chaque bouton correspond à une commande d'une allowlist stricte, avec seulement ses options typées ; le serveur reconstruit un argv fixe (sans shell), ledgerise chaque exécution (console.exec) et streame la sortie ci-dessous. Les commandes à effet d'état (upgrade) exigent une confirmation explicite. Les secrets (--passphrase-env) sont résolus côté serveur — vous ne fournissez qu'un nom de variable, jamais la valeur.
